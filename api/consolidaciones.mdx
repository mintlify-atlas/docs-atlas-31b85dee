All consolidation endpoints require a valid JWT token in the `Authorization` header (`Bearer <token>`). Consolidations represent monthly double-entry ledger records for a client. Each record is stored in either `consolidaciones_generales` or `consolidaciones_hoteles` depending on the value of `tipoRubro` supplied in the request. Non-admin users can only retrieve their own records; admin users receive records across all users. Every write operation is automatically registered in the system audit log ([Audit Logs](/api/logs)).

---

## POST /api/consolidaciones

Create a new consolidation record. The `tipoRubro` field determines which database table is targeted: `HOTELES` writes to `consolidaciones_hoteles`; any other value (including omission) defaults to `consolidaciones_generales`. `usuario_id` is automatically set from the authenticated JWT — it is not accepted from the request body.

**Auth:** Bearer token required.

### Request Body

<ParamField body="tipoRubro" type="string" required>
  Determines the target table. Accepted values: `GENERALES` (default) or `HOTELES`.
</ParamField>

<ParamField body="cliente_id" type="integer" required>
  ID of the client company this consolidation belongs to. Must reference an existing, active client. See [Clientes API](/api/clientes).
</ParamField>

<ParamField body="fecha_inicio" type="string" required>
  Period start date in `YYYY-MM-DD` format (e.g. `"2024-01-01"`).
</ParamField>

<ParamField body="fecha_fin" type="string" required>
  Period end date in `YYYY-MM-DD` format (e.g. `"2024-01-31"`).
</ParamField>

<ParamField body="observaciones" type="string">
  Optional free-text remarks for this consolidation period.
</ParamField>

## GET /api/consolidaciones

List consolidation records from both `consolidaciones_generales` and `consolidaciones_hoteles` (or either table when `tipo` is specified). Results are sorted by `fecha_creacion` descending. Non-admin users receive only their own records.

**Auth:** Bearer token required.

### Query Parameters

<ParamField query="cliente_id" type="integer">
  Filter records to a specific client ID.
</ParamField>

<ParamField query="fecha_desde" type="string">
  Inclusive lower bound on `fecha_inicio` (`YYYY-MM-DD`).
</ParamField>

<ParamField query="fecha_hasta" type="string">
  Inclusive upper bound on `fecha_fin` (`YYYY-MM-DD`).
</ParamField>

<ParamField query="tipo" type="string">
  Restrict results to one table. Accepted values: `GENERALES` or `HOTELES`. Omit to return records from both tables.
</ParamField>

### Response

<ResponseField name="message" type="string">
  `"Consolidaciones obtenidas exitosamente"`
</ResponseField>

<ResponseField name="data" type="array">
  Array of consolidation objects. Each element contains all 110 account fields plus joined columns `cliente_nombre`, `cliente_rtn`, `usuario_nombre`, computed `total_debe`, `total_haber`, and the injected `tipo` discriminator.
</ResponseField>

<ResponseField name="total" type="integer">
  Total number of records in the response array.
</ResponseField>

### Example

```bash
curl -X GET "https://api.example.com/api/consolidaciones?cliente_id=7&tipo=GENERALES&fecha_desde=2024-01-01&fecha_hasta=2024-12-31" \
  -H "Authorization: Bearer $TOKEN"
```

## PUT /api/consolidaciones/:id

Update an existing consolidation. Both `tipoRubro` and `motivo` are required in the request body. The server compares each submitted field against the current database value and rejects the request if no actual changes are detected. Every changed field is individually logged to `BitacoraCambios` (see [Audit Logs](/api/logs)).

**Auth:** Bearer token required.

<Warning>
  `tipoRubro` and `motivo` are required. The update will be rejected with `400` if either is missing, or if the submitted data is identical to the current record.
</Warning>

### Path Parameters

<ParamField path="id" type="integer" required>
  Numeric ID of the consolidation to update.
</ParamField>

### Request Body

<ParamField body="tipoRubro" type="string" required>
  Must be `GENERALES` or `HOTELES` — identifies the target table.
</ParamField>

<ParamField body="motivo" type="string" required>
  Free-text reason for the edit. Stored verbatim in the audit log entry for each changed field.
</ParamField>

<ParamField body="fecha_inicio" type="string">
  New period start date. Optional — only include fields you want to change.
</ParamField>

<ParamField body="fecha_fin" type="string">
  New period end date.
</ParamField>

<ParamField body="observaciones" type="string">
  Updated remarks.
</ParamField>

<ParamField body="[account]_debe" type="number">
  Updated debit value for any of the 55 account fields. Fields ending in `_debe` or `_haber` are coerced to `DECIMAL(18,2)`; empty strings and `null` are treated as `0`.
</ParamField>

<ParamField body="[account]_haber" type="number">
  Updated credit value for any of the 55 account fields.
</ParamField>

### Response

<ResponseField name="message" type="string">
  `"Consolidación actualizada exitosamente"`
</ResponseField>

<ResponseField name="data" type="object">
  Full updated consolidation record as re-fetched from the database after the update.
</ResponseField>

<ResponseField name="changesCount" type="integer">
  Number of individual fields that were changed. Each changed field generates one `BitacoraCambios` record.
</ResponseField>

## DELETE /api/consolidaciones/:id

Soft-delete a consolidation record. The row is **not removed** from the database — instead, `activo` is set to `0` and `fecha_actualizacion` is stamped with the current server time (`GETDATE()`). The record will no longer appear in any `GET` responses, but all data is preserved for audit purposes.

**Auth:** Bearer token required.

<Note>
  This is a **soft delete**. Data is preserved in the database with `activo = 0`. There is no hard-delete endpoint for consolidations. To recover a soft-deleted record, contact your database administrator.
</Note>

### Path Parameters

<ParamField path="id" type="integer" required>
  Numeric ID of the consolidation to deactivate.
</ParamField>

### Query Parameters

<ParamField query="tipo" type="string" required>
  Which table to target: `GENERALES` or `HOTELES`. This parameter is required — the endpoint returns `400` if omitted.
</ParamField>

### Response

<ResponseField name="message" type="string">
  `"Consolidación eliminada exitosamente"`
</ResponseField>

<ResponseField name="data" type="object">
  Confirmation object returned by the model's `delete()` method: `{ "message": "Consolidación eliminada exitosamente" }`.
</ResponseField>

## Error Responses

All endpoints return a consistent error envelope on failure:

| HTTP Status | Meaning |
|---|---|
| `400` | Missing or invalid request parameters (e.g. missing `tipo`, no changes detected) |
| `404` | Consolidation or referenced client not found |
| `500` | Unhandled server error; `details` field contains the original error message |

```json
{
  "error": "Parámetro tipo requerido (GENERALES o HOTELES)",
}
```

```json
{
  "error": "Error interno del servidor",
  "details": "Cannot read properties of null"
}
```
